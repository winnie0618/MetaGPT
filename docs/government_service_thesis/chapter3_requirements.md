# 第3章 面向政务服务流程的需求分析

## 3.1 应用场景

本文选取高校毕业生就业创业补贴办理作为实验场景。该场景具有政策文本明确、材料要求清晰、办理流程相对固定、公众咨询频繁等特点，同时也包含资格判断、补贴金额、审批结果和申诉复核等高风险问题，适合验证可追溯多智能体协同方法。

## 3.2 用户需求

用户主要希望系统能够回答申请对象、申请条件、所需材料、线上线下办理流程、补正材料、办理时限、审核公示和补贴发放等问题。对于复杂问题，用户还希望系统能够同时给出材料清单、办理步骤和政策依据。对于涉及审批结果、补贴金额或资格最终认定的问题，系统应明确提示需要人工复核。

## 3.3 功能需求

系统应支持政务服务问题输入、意图识别、政策依据检索、材料清单生成、流程步骤生成、风险等级判断、人工复核触发、结构化回答生成和追溯日志保存。每次回答应返回 `trace_id`，便于查询检索证据、执行动作、风险判断和最终回答。

## 3.4 非功能需求

系统应具备可解释性、可追溯性、可扩展性和降级可用性。缺少 RAG 依赖时，系统应回退到关键词检索；缺少 Web 依赖时，应给出清晰安装提示；缺少大语言模型时，应使用模板回答完成基本服务。

## 3.5 风险控制需求

政务服务场景涉及公共权益和行政责任，系统不得承诺审批结果、补贴金额、发放时间或最终资格认定。对于高风险事项，系统应触发人工复核提示，并说明回答仅作为政策依据和办理建议，不构成最终行政结论。
